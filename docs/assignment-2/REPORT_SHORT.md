## 1. Введение

### 1.1 Описание проекта

В рамках данного задания была выбрана предметная область "Library Stroll" — социальная платформа для художников и дизайнеров. Платформа предоставляет возможности публикации работ, организации коллекций, социального взаимодействия через комментарии и лайки, категоризации через теги, модерации контента и управления правами доступа.

Выбор данной предметной области обусловлен её сложностью и наличием различных типов функциональности, что позволяет продемонстрировать различия в подходах к разработке при использовании разных моделей жизненного цикла.

### 1.2 Основные компоненты системы

Основные компоненты системы Library Stroll: Frontend (Vue.js 3 + Inertia.js), Backend (Laravel 11), База данных (MySQL/SQLite), Файловое хранилище (Spatie Media Library), Real-time коммуникация (Pusher/Reverb), Очереди задач (Laravel Queue), Аутентификация (Laravel Sanctum + Fortify).

### 1.3 Цель работы

Целью работы является сравнительное моделирование архитектуры информационной системы Library Stroll в трех моделях жизненного цикла: Каскадная (Waterfall), Спиральная (Spiral) и Agile/Iterative. Для каждой модели необходимо описать этапы, процессы, роли и артефакты в соответствии со стандартом ISO/IEC 12207, разработать UML-диаграммы и провести сравнительный анализ.

## 2. Каскадная модель (Waterfall)

### 2.1 Описание модели

Каскадная модель представляет собой последовательную модель жизненного цикла, где каждый этап выполняется строго после завершения предыдущего. Изменения требований возможны только на ранних этапах проекта.

### 2.2 Этапы жизненного цикла (ISO/IEC 12207)

#### Этап 1: Анализ требований (4-6 недель)
**Процессы:** Процесс разработки требований, Процесс управления проектом  
**Роли:** Бизнес-аналитик, Системный аналитик, Заказчик  
**Артефакты:** SRS документ, Use Case диаграммы, ER-диаграммы, Техническое задание

#### Этап 2: Проектирование (6-8 недель)
**Процессы:** Процесс архитектурного проектирования, Процесс детального проектирования  
**Роли:** Архитектор системы, Database Designer, UX/UI дизайнер, Backend разработчик (senior)  
**Артефакты:** Архитектурная документация, Диаграммы компонентов и развертывания, Схема БД, API спецификация, Макеты интерфейсов

#### Этап 3: Реализация (12-16 недель)
**Процессы:** Процесс разработки, Процесс управления конфигурацией  
**Роли:** Backend разработчики (2-3), Frontend разработчики (2-3), DevOps инженер, QA инженер  
**Артефакты:** Исходный код, Миграции БД, Unit тесты, API endpoints, Vue компоненты

#### Этап 4: Тестирование (6-8 недель)
**Процессы:** Процесс тестирования, Процесс обеспечения качества  
**Роли:** QA инженеры (2-3), Тестировщики, Заказчик (UAT)  
**Артефакты:** Test планы, Test cases, Bug reports, Test отчеты

#### Этап 5: Развертывание (2-3 недели)
**Процессы:** Процесс развертывания, Процесс управления проектом  
**Роли:** DevOps инженер, Системный администратор, Разработчики  
**Артефакты:** Deployment план, Инструкции по развертыванию, Документация, Мониторинг

#### Этап 6: Сопровождение (постоянно)
**Процессы:** Процесс сопровождения  
**Роли:** Разработчики поддержки, DevOps инженер, Модераторы

## 3. Спиральная модель (Spiral)

### 3.1 Описание модели

Спиральная модель — итеративная модель с акцентом на управление рисками. Каждая итерация проходит через четыре квадранта: планирование, анализ рисков, разработка и оценка.

### 3.2 Итерации для Library Stroll

#### Итерация 1: Прототип (4-6 недель)
**Цели:** Прототип архитектуры, базовая аутентификация, публикация работ  
**Риски:** Неопределенность архитектуры, выбор технологий, производительность  
**Митигация:** Прототипирование, тестирование производительности, проверенные технологии

#### Итерация 2: Социальные функции (4-6 недель)
**Цели:** Комментарии, лайки, профили  
**Риски:** Производительность, безопасность (XSS), масштабируемость  
**Митигация:** Пагинация, санитизация, индексы БД

#### Итерация 3: Коллекции и теги (3-4 недели)
**Цели:** Коллекции, теги, поиск  
**Риски:** Производительность поиска, UX

#### Итерация 4: Мессенджер (6-8 недель)
**Цели:** Real-time мессенджер, группы, вложения  
**Риски:** Сложность real-time, масштабируемость WebSocket, безопасность  
**Митигация:** Pusher, нагрузочные тесты, шифрование

#### Итерация 5: Модерация (4-6 недель)
**Цели:** Система жалоб, панель модератора, админ-панель  
**Риски:** Злоупотребление, производительность, безопасность

#### Итерация 6: Оптимизация и релиз (4-6 недель)
**Цели:** Оптимизация, исправление багов, подготовка к релизу

## 4. Agile/Iterative модель

### 4.1 Описание модели

Agile модель — итеративная модель с короткими спринтами (2 недели), фокусом на быструю обратную связь и адаптацию к изменениям. Используется Scrum как основной фреймворк.

### 4.2 Спринты для Library Stroll

#### Sprint 0: Инфраструктура (1 неделя)
Настройка репозитория, CI/CD, dev окружения, базовой архитектуры

#### Sprint 1-2: Аутентификация и профили (4 недели)
Регистрация, вход, email верификация, профили пользователей

#### Sprint 3-4: Публикация работ (4 недели)
Загрузка изображений, создание и редактирование работ, отображение в галерее

#### Sprint 5-6: Комментарии и лайки (4 недели)
Система комментариев, ответы, лайки, уведомления

#### Sprint 7-8: Коллекции и теги (4 недели)
Создание коллекций, система тегов, поиск по тегам

#### Sprint 9-11: Мессенджер (6 недель)
Real-time мессенджер, групповые беседы, вложения, реакции

#### Sprint 12-13: Модерация (4 недели)
Система жалоб, панель модератора, административная панель

#### Sprint 14: Оптимизация и релиз (2 недели)
Оптимизация производительности, исправление багов, подготовка к production

## 6. Имитационное моделирование изменений требований

### 6.1 Сценарий: Добавление функции "Система подписок"

В качестве сценария было выбрано добавление функции "Система подписок" через 6 месяцев после начала проекта.

### 6.2 Влияние на модели

**Waterfall модель:**
- Текущее состояние: Завершены анализ и проектирование, реализация на 50%
- Процесс: Возврат к анализу (2 недели, $6,000), обновление проектирования (3 недели, $9,000), доработка реализации (4-5 недель, $12,000-$15,000)
- Время: 8-10 недель
- Стоимость: $24,000-$30,000
- Устойчивость: Низкая (1/5)

**Spiral модель:**
- Текущее состояние: Завершены итерации 1-2, итерация 3 на 50%
- Процесс: Анализ рисков (1 неделя, $3,000), включение в следующую итерацию (3-5 недель, $9,000-$15,000)
- Время: 4-6 недель
- Стоимость: $12,000-$18,000
- Устойчивость: Средняя (3/5)

**Agile модель:**
- Текущее состояние: Завершены Sprint 1-6, Sprint 7 на 50%
- Процесс: Добавление в backlog (1 день, $200), включение в следующий спринт (1-2 недели, $2,800-$5,800)
- Время: 1-2 недели
- Стоимость: $3,000-$6,000
- Устойчивость: Высокая (5/5)

### 6.3 Выводы по устойчивости

Результаты показывают, что Agile модель демонстрирует наивысшую устойчивость к изменениям (5/5) с возможностью внесения изменений за 1-2 недели и стоимостью $3,000-$6,000. Waterfall модель показывает наименьшую устойчивость (1/5), требуя 8-10 недель и $24,000-$30,000.

### 7.2 Обоснованный выбор оптимальной модели

Для проекта Library Stroll рекомендуется использование **Agile/Scrum модели**.

**Обоснование:**

1. **Характеристики проекта:** Library Stroll — социальная платформа, требующая быстрой обратной связи от пользователей, итеративной разработки real-time функций и адаптации к изменяющимся требованиям.

2. **Преимущества Agile:**
   - Высокая гибкость (5/5) — быстрая адаптация к пользовательским запросам
   - Быстрая разработка (5/5) — 7 месяцев до релиза vs 12 месяцев в Waterfall
   - Низкий риск (4/5) — короткие спринты минимизируют потери
   - Высокая устойчивость к изменениям (5/5) — изменения за 1-2 недели
   - Эффективная стоимость (4/5) — $176,000 (самая низкая)

3. **Недостатки других моделей:**
   - Waterfall: Низкая гибкость (2/5), долгий цикл (12 месяцев), низкая устойчивость (1/5)
   - Spiral: Средняя гибкость (4/5), требует опытных специалистов, выше стоимость ($188,000)

### 7.3 Рекомендации по внедрению Agile

**Структура команды:** Product Owner (1), Scrum Master (1), Backend разработчики (2-3), Frontend разработчики (2-3), QA инженер (1-2), DevOps инженер (1), UX/UI дизайнер (1).

**Ключевые практики:** Sprint Planning, Daily Standups, Sprint Review, Sprint Retrospective, Continuous Integration, Continuous Deployment.

**Метрики успеха:** Velocity 15-20 story points за спринт, Code Coverage минимум 70%, Deployment Frequency после каждого спринта.

### 7.4 Заключение

На основе сравнительного анализа трех моделей жизненного цикла для проекта Library Stroll сделан вывод о том, что **Agile/Scrum модель** является оптимальным выбором благодаря максимальной гибкости, быстрой разработке, низкому риску и высокой устойчивости к изменениям требований.

**Итоговая рекомендация:** Использовать **Agile/Scrum** с 2-недельными спринтами для разработки Library Stroll.
