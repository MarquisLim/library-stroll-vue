## 1. Введение

### 1.1 Описание проекта

В рамках данного задания была выбрана предметная область "Library Stroll" — социальная платформа для художников и дизайнеров. Данная платформа предоставляет следующие возможности:

- Публикация и управление художественными работами (изображения, видео, GIF)
- Организация работ в коллекции
- Социальное взаимодействие через комментарии, лайки и сообщения
- Категоризация через систему тегов
- Модерация контента через систему жалоб
- Управление правами доступа и приватностью

Выбор данной предметной области обусловлен её сложностью и наличием различных типов функциональности, что позволяет продемонстрировать различия в подходах к разработке при использовании разных моделей жизненного цикла.

### 1.2 Основные компоненты системы

В ходе анализа проекта Library Stroll были определены следующие основные компоненты системы:

- **Frontend:** Vue.js 3 + Inertia.js + Tailwind CSS
- **Backend:** Laravel 11 (PHP 8.2)
- **База данных:** MySQL/SQLite
- **Файловое хранилище:** Spatie Media Library
- **Real-time коммуникация:** Pusher/Reverb
- **Очереди задач:** Laravel Queue
- **Аутентификация:** Laravel Sanctum + Fortify

### 1.3 Цель работы

Целью данной работы является сравнительное моделирование архитектуры информационной системы Library Stroll в трех различных моделях жизненного цикла:

1. Каскадная модель (Waterfall)
2. Спиральная модель (Spiral)
3. Agile/Iterative модель

Для каждой модели необходимо описать этапы, процессы, роли и артефакты в соответствии со стандартом ISO/IEC 12207, разработать UML-диаграммы (активностей, компонентов, развертывания) и провести сравнительный анализ по критериям гибкости, производительности и управляемости.

## 2. Каскадная модель (Waterfall)

### 2.1 Описание модели

Каскадная модель представляет собой последовательную модель жизненного цикла разработки программного обеспечения, где каждый этап выполняется строго после завершения предыдущего. Изменения требований возможны только на ранних этапах проекта, что обеспечивает четкую структуру и предсказуемость процесса разработки.

### 2.2 Этапы жизненного цикла (ISO/IEC 12207)

В соответствии со стандартом ISO/IEC 12207, для проекта Library Stroll была спроектирована следующая последовательность этапов каскадной модели.

#### Этап 1: Анализ требований (Requirements Analysis)

**Длительность:** 4-6 недель

**Процессы (ISO/IEC 12207):**
- Процесс разработки требований (Requirements Development Process)
- Процесс управления проектом (Project Management Process)

**Роли:**
- Бизнес-аналитик — сбор и анализ требований заказчика
- Системный аналитик — формализация требований в техническую документацию
- Заказчик/Product Owner — предоставление требований и их утверждение

**Артефакты:**
- Документ требований (SRS - Software Requirements Specification)
- Use Case диаграммы
- ER-диаграммы (из задания 1)
- Техническое задание

**Для Library Stroll были определены следующие ключевые требования:**
- Требования к публикации работ (загрузка, редактирование, удаление)
- Требования к системе комментариев (комментарии, ответы, модерация)
- Требования к мессенджеру (диалоги, группы, real-time коммуникация)
- Требования к модерации контента (система жалоб, панель модератора)

#### Этап 2: Проектирование (System Design)

**Длительность:** 6-8 недель

**Процессы:**
- Процесс архитектурного проектирования (Architectural Design Process)
- Процесс детального проектирования (Detailed Design Process)

**Роли:**
- Архитектор системы — проектирование общей архитектуры
- Database Designer — проектирование схемы базы данных
- UX/UI дизайнер — проектирование пользовательского интерфейса
- Backend разработчик (senior) — проектирование API и бизнес-логики

**Артефакты:**
- Архитектурная документация
- Диаграммы компонентов
- Диаграммы развертывания
- Схема базы данных
- API спецификация (OpenAPI/Swagger)
- Макеты интерфейсов (Wireframes, Mockups)

На данном этапе была спроектирована архитектура системы Library Stroll с использованием Laravel в качестве backend-фреймворка и Vue.js для frontend-части. База данных включает 10 основных таблиц, связанных между собой через внешние ключи и промежуточные таблицы для many-to-many связей.

#### Этап 3: Реализация (Implementation)

**Длительность:** 12-16 недель

**Процессы:**
- Процесс разработки (Development Process)
- Процесс управления конфигурацией (Configuration Management Process)

**Роли:**
- Backend разработчики (2-3) — разработка Laravel приложения
- Frontend разработчики (2-3) — разработка Vue.js компонентов
- DevOps инженер — настройка инфраструктуры
- QA инженер — начало тестирования на этапе реализации

**Артефакты:**
- Исходный код приложения
- Миграции базы данных
- Unit тесты
- API endpoints
- Компоненты Vue.js

На этапе реализации выполняется разработка всех компонентов системы согласно архитектурной документации, созданной на предыдущем этапе. Все изменения в коде документируются и проходят code review.

#### Этап 4: Тестирование (Testing)

**Длительность:** 6-8 недель

**Процессы:**
- Процесс тестирования (Testing Process)
- Процесс обеспечения качества (Quality Assurance Process)

**Роли:**
- QA инженеры (2-3) — разработка тестовых сценариев и выполнение тестирования
- Тестировщики — функциональное и интеграционное тестирование
- Заказчик — участие в User Acceptance Testing (UAT)

**Артефакты:**
- Test планы
- Test cases
- Bug reports
- Test отчеты

Тестирование включает unit тестирование отдельных компонентов, integration тестирование взаимодействия компонентов, system тестирование всей системы и user acceptance testing с участием заказчика.

#### Этап 5: Развертывание (Deployment)

**Длительность:** 2-3 недели

**Процессы:**
- Процесс развертывания (Deployment Process)
- Процесс управления проектом

**Роли:**
- DevOps инженер — подготовка production окружения
- Системный администратор — настройка серверов и сетевой инфраструктуры
- Разработчики — поддержка при развертывании

**Артефакты:**
- Deployment план
- Инструкции по развертыванию
- Документация для пользователей
- Мониторинг и логи

На этапе развертывания выполняется миграция данных, настройка production окружения, развертывание приложения и настройка систем мониторинга.

#### Этап 6: Сопровождение (Maintenance)

**Длительность:** Постоянно

**Процессы:**
- Процесс сопровождения (Maintenance Process)

**Роли:**
- Разработчики поддержки — исправление ошибок и обновления
- DevOps инженер — поддержка инфраструктуры
- Модераторы контента — модерация пользовательского контента

Диаграмма развертывания показывает физическую архитектуру системы Library Stroll. Все компоненты развертываются одновременно после завершения этапа тестирования.

Основные узлы развертывания:
- **DMZ Zone:** Load Balancer (Nginx) для распределения нагрузки

### 2.6 Характеристики модели

**Преимущества каскадной модели:**
- Четкая структура и планирование проекта
- Подробная документация на каждом этапе
- Легко управлять бюджетом и сроками
- Подходит для проектов с четкими, неизменными требованиями

**Недостатки каскадной модели:**
- Изменения требований сложно вносить после начала реализации
- Долгий цикл разработки (12 месяцев до первого релиза)
- Риск устаревания требований к моменту релиза
- Позднее обнаружение проблем, что увеличивает стоимость их исправления

**Временная шкала:** Общая длительность разработки составляет приблизительно 12 месяцев до первого релиза.

## 3. Спиральная модель (Spiral)

### 3.1 Описание модели

Спиральная модель представляет собой итеративную модель жизненного цикла с акцентом на управление рисками. Каждая итерация проходит через четыре квадранта: планирование, анализ рисков, разработка и оценка. Данная модель позволяет раннее обнаруживать проблемы и управлять рисками на каждой итерации.

### 3.2 Структура спирали

Каждая итерация спиральной модели включает четыре квадранта:

1. **Квадрант 1: Планирование** — определение целей итерации, ограничений и альтернатив
2. **Квадрант 2: Анализ рисков** — идентификация, оценка и планирование митигации рисков
3. **Квадрант 3: Разработка и тестирование** — создание функциональности и её проверка
4. **Квадрант 4: Оценка и планирование** — анализ результатов итерации и планирование следующей

### 3.3 Итерации для Library Stroll

Для проекта Library Stroll была спроектирована последовательность из шести итераций, каждая из которых фокусируется на определенной функциональности и управлении связанными с ней рисками.

#### Итерация 1: Прототип и базовая функциональность (4-6 недель)

**Цели итерации:**
- Создание прототипа архитектуры системы
- Реализация базовой аутентификации пользователей
- Реализация публикации художественных работ

**Идентифицированные риски:**
- Неопределенность в выборе архитектуры
- Выбор технологий для реализации
- Производительность загрузки файлов

**Митигация рисков:**
- Прототипирование ключевых компонентов на раннем этапе
- Тестирование производительности загрузки файлов
- Выбор проверенных технологий (Laravel, Vue.js)

**Процессы (ISO/IEC 12207):**
- Процесс разработки (Development Process)
- Процесс управления рисками (Risk Management Process)
- Процесс обеспечения качества (Quality Assurance Process)

**Роли:** Архитектор, Backend разработчик (1), Frontend разработчик (1), QA инженер

**Артефакты:** Прототип архитектуры, базовая аутентификация, загрузка и отображение работ, отчет об оценке рисков

#### Итерация 2: Социальные функции (4-6 недель)

**Цели итерации:**
- Система комментариев с возможностью ответов
- Система лайков для работ
- Профили пользователей

**Идентифицированные риски:**
- Производительность при большом количестве комментариев
- Безопасность (риск XSS атак в комментариях)
- Масштабируемость системы

**Митигация рисков:**
- Реализация пагинации комментариев
- Санитизация всех входных данных
- Создание индексов базы данных для оптимизации запросов

**Процессы:**
- Процесс разработки
- Процесс тестирования (Testing Process)
- Процесс управления конфигурацией (Configuration Management Process)

**Роли:** Backend разработчики (2), Frontend разработчики (2), QA инженер, Security специалист

#### Итерация 3: Коллекции и теги (3-4 недели)

**Цели итерации:**
- Система коллекций для организации работ
- Система тегов для категоризации
- Поиск по тегам

**Идентифицированные риски:**
- Производительность поиска по тегам
- UX организации коллекций

**Митигация рисков:**
- Реализация полнотекстового поиска
- Кэширование популярных тегов
- User testing интерфейса коллекций

#### Итерация 4: Мессенджер (6-8 недель)

**Цели итерации:**
- Real-time мессенджер между пользователями
- Групповые беседы
- Вложения в сообщениях

**Идентифицированные риски:**
- Сложность реализации real-time коммуникации
- Масштабируемость WebSocket соединений
- Безопасность сообщений

**Митигация рисков:**
- Использование проверенного решения (Pusher)
- Тестирование системы под нагрузкой
- Шифрование сообщений

**Процессы:**
- Процесс разработки
- Процесс управления рисками
- Процесс тестирования

**Роли:** Backend разработчики (2), Frontend разработчики (2), DevOps инженер, QA инженер

#### Итерация 5: Модерация и администрирование (4-6 недель)

**Цели итерации:**
- Система жалоб на контент и пользователей
- Панель модератора для обработки жалоб
- Административная панель
- Блокировка контента

**Идентифицированные риски:**
- Злоупотребление системой жалоб
- Производительность обработки жалоб
- Безопасность административной панели

**Митигация рисков:**
- Ограничения на количество жалоб от одного пользователя
- Использование очередей для обработки жалоб
- Строгие права доступа к административной панели

#### Итерация 6: Оптимизация и релиз (4-6 недель)

**Цели итерации:**
- Оптимизация производительности системы
- Исправление критических багов
- Подготовка к релизу
- Создание документации

**Идентифицированные риски:**
- Производительность системы под нагрузкой
- Критические баги, обнаруженные на поздних этапах

**Митигация рисков:**
- Нагрузочное тестирование системы
- Регрессионное тестирование всех функций
- Резервное время на исправления

**Процессы:**
- Процесс обеспечения качества
- Процесс развертывания (Deployment Process)
- Процесс документирования (Documentation Process)

   - Планирование спринта с новой функцией
   - Разработка функции подписок в текущем или следующем спринте
   - Непрерывная интеграция

- Низкий риск — функция легко интегрируется благодаря модульной архитектуре

### 7.2 Обоснованный выбор оптимальной модели

На основе проведенного сравнительного анализа и имитационного моделирования для проекта Library Stroll рекомендуется использование **Agile/Scrum модели**.

#### 7.2.1 Характеристики проекта Library Stroll

Library Stroll представляет собой социальную платформу со следующими особенностями:
- Социальные функции требуют быстрой обратной связи от пользователей
- Real-time коммуникация лучше разрабатывается итеративно
- Пользовательский контент требует адаптации к запросам пользователей
- Высокая неопределенность требований на начальном этапе

#### 7.2.2 Преимущества Agile для Library Stroll

**Высокая гибкость (5/5)**
Социальные платформы требуют быстрой адаптации к пользовательским запросам. Требования могут меняться на основе аналитики и обратной связи пользователей. Короткие спринты (2 недели) позволяют быстро вносить изменения в соответствии с новыми требованиями.

**Быстрая разработка (5/5)**
Agile модель обеспечивает 7 месяцев до первого релиза по сравнению с 12 месяцами в Waterfall модели. Это позволяет раннее получать обратную связь от пользователей и запускать MVP уже через 2-3 месяца.

**Низкий риск (4/5)**
Короткие спринты минимизируют риск больших потерь. Раннее обнаружение проблем через непрерывную интеграцию позволяет быстро реагировать на проблемы. Возможность остановки проекта на любом спринте снижает финансовые риски.

**Высокая устойчивость к изменениям (5/5)**
Изменения требований можно внести за 1-2 недели с низкой стоимостью ($3,000-$6,000 по сравнению с $24,000-$30,000 в Waterfall). Минимальное влияние на сроки проекта делает Agile модель оптимальной для проектов с изменяющимися требованиями.

**Эффективная стоимость (4/5)**
Общая стоимость $176,000 является самой низкой среди всех моделей. Раннее обнаружение проблем экономит ресурсы. Возможность остановки на любом этапе снижает финансовые риски.

#### 7.2.3 Почему другие модели менее подходят

**Waterfall модель:**
Низкая гибкость (2/5) означает, что изменения требований критически влияют на проект. Долгий цикл разработки (12 месяцев) увеличивает риск устаревания требований. Низкая устойчивость к изменениям (1/5) делает Waterfall модель непригодной для социальных платформ с изменяющимися требованиями.

**Spiral модель:**
Средняя гибкость (4/5) лучше чем Waterfall, но хуже чем Agile. Требует опытных специалистов по управлению рисками, что может быть ограничением для команды. Более высокая стоимость ($188,000) и средняя устойчивость к изменениям (3/5) делают Spiral модель менее предпочтительной по сравнению с Agile.

### 7.3 Рекомендации по внедрению Agile

#### 7.3.1 Структура команды

Для успешной реализации Agile подхода рекомендуется следующая структура команды:

- **Product Owner** (1) — управление backlog и приоритетами
- **Scrum Master** (1) — фасилитация процесса
- **Backend разработчики** (2-3) — Laravel разработка
- **Frontend разработчики** (2-3) — Vue.js разработка
- **QA инженер** (1-2) — тестирование
- **DevOps инженер** (1) — инфраструктура и CI/CD
- **UX/UI дизайнер** (1) — дизайн интерфейсов

#### 7.3.2 Ключевые практики

Для успешной реализации Agile подхода необходимо внедрить следующие практики:

1. **Sprint Planning** — планирование задач на 2 недели с оценкой сложности
2. **Daily Standups** — ежедневные 15-минутные встречи для синхронизации команды
3. **Sprint Review** — демонстрация результатов заказчику
4. **Sprint Retrospective** — анализ процесса и выявление возможностей для улучшения
5. **Continuous Integration** — автоматическая интеграция кода после каждого коммита
6. **Continuous Deployment** — автоматическое развертывание после успешного прохождения тестов

#### 7.3.3 Метрики успеха

Для отслеживания эффективности Agile подхода рекомендуется использовать следующие метрики:

- **Velocity:** 15-20 story points за спринт
- **Sprint Burndown:** отслеживание прогресса выполнения задач в спринте
- **Code Coverage:** минимум 70% покрытия тестами
- **Deployment Frequency:** развертывание после каждого спринта
- **Mean Time to Recovery:** быстрое восстановление при обнаружении проблем

### 7.4 Заключение

На основе проведенного сравнительного анализа трех моделей жизненного цикла (Waterfall, Spiral, Agile) для проекта Library Stroll был сделан вывод о том, что **Agile/Scrum модель** является оптимальным выбором по следующим причинам:

1. Максимальная гибкость к изменениям требований
2. Быстрая разработка и ранняя обратная связь
3. Низкий риск благодаря коротким спринтам
4. Эффективная стоимость разработки
5. Высокая устойчивость к изменениям

Проект Library Stroll, как социальная платформа, требует итеративной разработки с быстрой адаптацией к пользовательским запросам, что идеально соответствует принципам Agile модели.

**Итоговая рекомендация:** Использовать **Agile/Scrum** с 2-недельными спринтами для разработки Library Stroll.
