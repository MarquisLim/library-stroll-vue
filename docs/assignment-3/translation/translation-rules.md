# Правила трансляции DFD → UML

## Описание

Данный документ описывает формальные правила трансляции элементов DFD в элементы UML.

## Формальные правила трансляции

### Правило 1: Трансформация процесса в класс

**Условие:** Процесс имеет имя, входы, выходы и может иметь состояние

**Трансформация:**
```
DFD: Процесс "N.Имя" (входы: I1, I2, выходы: O1, O2)
  ↓
UML: class Имя {
    +метод1(I1): O1
    +метод2(I2): O2
}
```

**Пример:**
```
DFD: Процесс "2.1 Валидация файла" (вход: Файл, выход: Результат валидации)
  ↓
UML: class FileValidator {
    +validate(file: File): ValidationResult
}
```

### Правило 2: Трансформация процесса в метод

**Условие:** Процесс является элементарной операцией без состояния

**Трансформация:**
```
DFD: Процесс "N.Имя" (вход: I, выход: O)
  ↓
UML: метод в соответствующем классе:
    +имя(I): O
```

**Пример:**
```
DFD: Процесс "2.2 Обработка изображения" (вход: Изображение, выход: Обработанное)
  ↓
UML: class ImageProcessor {
    +process(image: Image): ProcessedImage
}
```

### Правило 3: Трансформация потока данных

**Трансформация:**
```
DFD: Поток данных "Имя_потока" от Процесс1 к Процесс2
  ↓
UML: 
  - Если Процесс1 → Процесс2: параметр метода
  - Если Процесс1 → Хранилище: параметр метода сохранения
  - Если Хранилище → Процесс2: возвращаемое значение метода чтения
```

**Пример:**
```
DFD: Поток "Файл" от Пользователь к "2.1 Валидация"
  ↓
UML: FileValidator.validate(file: File)
```

### Правило 4: Трансформация хранилища данных

**Трансформация:**
```
DFD: Хранилище данных "Имя_хранилища"
  ↓
UML: class ИмяХранилищаRepository {
    +save(data: Data): void
    +findById(id: int): Data
    +update(id: int, data: Data): void
    +delete(id: int): void
}
```

**Пример:**
```
DFD: Хранилище "База данных" (artworks)
  ↓
UML: class ArtworkRepository {
    +create(artwork: Artwork): Artwork
    +findById(id: int): Artwork
    +update(id: int, artwork: Artwork): Artwork
    +delete(id: int): void
}
```

### Правило 5: Трансформация внешней сущности

**Трансформация:**
```
DFD: Внешняя сущность "Имя"
  ↓
UML: 
  - class Имя (если это система)
  - actor Имя (если это пользователь)
```

**Пример:**
```
DFD: Внешняя сущность "Пользователи"
  ↓
UML: actor User
```

### Правило 6: Трансформация последовательности процессов

**Трансформация:**
```
DFD: Процесс1 → Процесс2 → Процесс3
  ↓
UML: Sequence Diagram:
    Объект1: метод1()
    Объект2: метод2()
    Объект3: метод3()
```

**Пример:**
```
DFD: 2.1 Валидация → 2.2 Обработка → 2.3 Сохранение
  ↓
UML: Sequence Diagram:
    FileValidator: validate(file)
    ImageProcessor: process(image)
    StorageService: save(file, path)
```

### Правило 7: Трансформация условий

**Трансформация:**
```
DFD: Процесс → {Условие?} → Процесс1 / Процесс2
  ↓
UML: Activity Diagram:
    [Условие] → Действие1
    [не Условие] → Действие2
```

**Пример:**
```
DFD: Валидация → {Файл валиден?} → Обработка / Ошибка
  ↓
UML: Activity Diagram:
    [file.isValid()] → processImage()
    [not file.isValid()] → showError()
```

## Алгоритм трансформации

### Шаг 1: Идентификация процессов
1. Пройти по всем процессам DFD
2. Определить тип каждого процесса (класс или метод)
3. Создать соответствующие UML элементы

### Шаг 2: Трансформация потоков данных
1. Для каждого потока данных определить источник и назначение
2. Преобразовать в параметры методов или возвращаемые значения
3. Определить типы данных

### Шаг 3: Трансформация хранилищ
1. Для каждого хранилища создать класс Repository
2. Добавить методы CRUD операций
3. Определить связи с процессами

### Шаг 4: Создание диаграммы классов
1. Объединить все классы
2. Добавить ассоциации между классами
3. Добавить зависимости

### Шаг 5: Создание диаграммы последовательности
1. Для каждой цепочки процессов создать диаграмму последовательности
2. Определить объекты и их взаимодействие
3. Добавить сообщения между объектами

### Шаг 6: Создание диаграммы активности
1. Преобразовать последовательность процессов в действия
2. Добавить условия и ветвления
3. Добавить начальные и конечные узлы
