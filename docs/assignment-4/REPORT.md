## 2. Архитектура системы

### 2.1 Общая архитектура

Система управления метаданными построена по принципу DSM (Domain-Specific Modeling) и состоит из следующих слоев:

1. **Frontend Layer** - динамические компоненты Vue.js
2. **API Layer** - REST API для работы с метаданными
3. **Interpreter Layer** - интерпретатор метамодели
4. **Metadata Layer** - загрузчики метаданных
5. **Database Layer** - база данных метаданных и данных приложения

### 2.2 Хранение метаданных

Метаданные хранятся в базе данных MySQL в следующих таблицах:

- `meta_entities` - сущности
- `meta_attributes` - атрибуты сущностей
- `meta_relationships` - связи между сущностями
- `meta_forms` - конфигурации форм
- `meta_tables` - конфигурации таблиц

## 3. Онтологическая метамодель

### 3.1 Структура метамодели

Онтологическая метамодель определяет структуру описания метаданных для динамической генерации интерфейсов.

**Основные концепты:**

1. **Entity (Сущность)** - описание сущности предметной области
2. **Attribute (Атрибут)** - описание атрибута сущности
3. **Relationship (Связь)** - описание связи между сущностями
4. **Form (Форма)** - описание формы для редактирования сущности
5. **Table (Таблица)** - описание таблицы для отображения списка

### 3.2 Пример метамодели

```json
{
  "entities": [
    {
      "id": "Artwork",
      "name": "Artwork",
      "label": "Художественная работа",
      "attributes": [
        {
          "id": "title",
          "name": "title",
          "label": "Название",
          "type": "string",
          "required": true
        }
      ]
    }
  ]
}
```

**Полное описание метамодели:** см. `metamodel/ontology.md`

## 4. Интерпретатор метамодели

### 4.1 Назначение интерпретатора

Интерпретатор метамодели (`MetamodelInterpreter`) отвечает за:

1. Загрузку метаданных из базы данных
2. Генерацию конфигураций форм на основе метаданных
3. Генерацию конфигураций таблиц на основе метаданных
4. Валидацию данных форм на основе метаданных

### 4.2 Основные методы

- `loadEntity($entityId)` - загрузка полных метаданных сущности
- `generateFormConfig($entityId, $formName)` - генерация конфигурации формы
- `generateTableConfig($entityId, $tableName)` - генерация конфигурации таблицы
- `validateFormData($entityId, $formName, $data)` - валидация данных формы
- `clearCache($entityId)` - очистка кэша метаданных

**Код интерпретатора:** см. `interpreter/interpreter.php`

## 6. Эксперимент с изменением метамодели

### 6.1 Цель эксперимента

Продемонстрировать способность системы адаптироваться к изменению метамодели без перекомпиляции приложения.

### 6.2 Сценарий эксперимента

#### Шаг 1: Исходное состояние

Система имеет метаданные для сущности Artwork с атрибутами:
- id, title, description, file_path, created_at

**Форма создания содержит:** title, description, file_path, user

#### Шаг 2: Изменение метамодели

Добавлен новый атрибут `tags` через SQL запрос:

```sql
INSERT INTO meta_attributes (...) VALUES ('artwork_tags', 'Artwork', 'tags', ...);
UPDATE meta_forms SET fields_config = ...; -- добавление поля в форму
UPDATE meta_tables SET columns_config = ...; -- добавление колонки в таблицу
```

#### Шаг 3: Результат

**После изменения:**
- Форма создания автоматически включает новое поле "Теги"
- Таблица списка автоматически показывает новую колонку "Теги"
- Никакой перекомпиляции не требуется

**Состояние ДО:** см. `experiment/before.md`  
**SQL изменения:** см. `experiment/change.sql`

### 6.4 Выводы эксперимента

1. ✅ **Система полностью адаптивна** - изменения метамодели сразу отражаются в интерфейсе
2. ✅ **Без перекомпиляции** - достаточно изменить данные в БД
3. ✅ **Динамическая генерация** - интерфейсы генерируются на основе метаданных
4. ✅ **Гибкость** - можно добавлять/изменять/удалять поля без изменения кода

## 7. Выводы

### 7.1 Преимущества DSM-подхода

1. **Гибкость:** Изменение метамодели не требует изменения кода приложения
2. **Быстрота разработки:** Новые формы и таблицы создаются через метаданные
3. **Единообразие:** Все интерфейсы генерируются по единым правилам
4. **Масштабируемость:** Легко добавлять новые сущности и атрибуты

### 7.2 Ограничения

1. **Производительность:** Требуется кэширование метаданных
2. **Сложность:** Сложные интерфейсы могут требовать расширения метамодели
3. **Валидация:** Сложные правила валидации могут быть ограничены метамоделью

### 7.3 Применимость

DSM-подход применим для:
- Систем с часто изменяющимися требованиями
- Административных панелей
- Систем управления контентом
- Прототипирования интерфейсов

### 7.4 Итоговые выводы

Реализованная система управления метаданными успешно демонстрирует принципы DSM-подхода:

- ✅ Метаданные хранятся в БД и могут изменяться динамически
- ✅ Интерфейсы автоматически генерируются на основе метаданных
- ✅ Система адаптируется к изменениям без перекомпиляции
- ✅ Онтологическая метамодель обеспечивает структурированное описание метаданных

Система готова к использованию и может быть расширена для поддержки более сложных сценариев.
